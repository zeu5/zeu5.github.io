Getting distributed time right

12th April 2024

---

As a note to the paper Time, Clocks and Ordering of Events in a Distributed System, Leslie Lamport says the following - "Special relativity teaches us that there is no invariant total ordering of events in space-time; different observers can disagree about which of two events happened first. There is only a partial order in which an event e1 precedes an event e2 iff e1 can causally affect e2.". When two nodes of a distributed system communicate with each other, they can disagree about each other's time since there is no global time. With realistic clocks that measure time very accurately, we can at best switch to a model where the time of the two processes have a bounded drift.

But why are we so concerned with time? An execution of a distributed system is not a single sequence of events that occur one after another but a partial order of events represented by a DAG. To capture an execution we record all the events and their causal ordering. Now you may ask why do we need to capture executions? So that we can replay them. Bugs in distributed systems require intricate ordering of events to appear and are therefore rare. To replay a bug, we need to capture the exact execution (partial order of events).
